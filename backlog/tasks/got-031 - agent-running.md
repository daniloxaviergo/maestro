## Implementation Plan: Agent Status Script

### 1. Technical Approach

The script will check agent status by:
1. Reading agent configurations from the `agents/` directory
2. For each enabled agent with a configured script:
   - Parse the agent's execution log file (e.g., `agents/{agent}/execution.log`)
   - Check the last log line for "Task processing complete" message
   - If found: agent is **idle** (last task completed)
   - If not found and log exists: agent is **running** (task in progress)
   - If no log exists: agent is **idle** (never run)
3. Output status in a human-readable format (optionally JSON for machine parsing)

**Design Decisions:**
- Log file path will be configurable in `config.yml` (extends existing AgentConfig)
- Default log path: `agents/{agent}/execution.log`
- Uses bash for portability and ease of maintenance (consistent with agent scripts)
- Supports both human-readable and JSON output modes
- Handles edge cases: missing logs, malformed logs, permission errors

### 2. Files to Modify

#### New Files Created:
- `scripts/agent_status.sh` - Main status checking script

#### Files to Modify:
- `pkg/config/types.go` - Add `LogPath` field to `AgentConfig` struct
- `pkg/config/config.go` - Update `LoadConfig` to handle new field

### 3. Dependencies

**Prerequisites:**
- bash 4.0+ (for associative arrays and advanced features)
- Standard Unix tools: `grep`, `sed`, `awk`, `tail`, `stat`
- Agent configurations must exist in `agents/` directory

**Existing Components:**
- Agent config format from `pkg/config/`
- Agent execution logging from existing agent scripts
- Task processing logs generated by agent scripts

### 4. Code Patterns

**Bash Script Conventions (matching `scripts/attach.sh`):**
- Use `set -euo pipefail` for strict error handling
- Helper functions: `error()`, `warning()`, `info()`, `debug()`
- Absolute paths via `SCRIPT_DIR` and `PROJECT_ROOT`
- Graceful degradation (don't fail on missing agent logs)
- fzf support for interactive selection (optional enhancement)

**Config Parsing:**
- Simple grep/sed for YAML parsing (no external dependencies)
- Fallback to default log path if not configured
- Validate script path exists before checking status

**Output Formats:**
- Human-readable: List of agents with status
- JSON: Machine-parsable output for automation
- Verbose: Include log file path and last timestamp

### 5. Testing Strategy

**Test Cases:**
1. **Agent idle** - Log exists with "Task processing complete" as last line
2. **Agent running** - Log exists but last line is NOT "Task processing complete"
3. **Agent never run** - No log file exists
4. **Agent disabled** - Config has `enabled: false` (should be skipped or marked differently)
5. **Malformed log** - Log exists but is empty or corrupted
6. **Missing script** - Script path configured but file doesn't exist

**Testing Approach:**
- Manual: Run monitor + trigger task assignment, then check status
- Script: Create test agents with controlled log content
- Verify JSON output can be parsed (e.g., `jq`)

**Unit Test Considerations:**
- Each test needs isolated agent directory
- Mock log files with controlled content
- Verify exit code and output format

### 6. Risks and Considerations

**Known Challenges:**
- **Log rotation**: If log files grow large, need to handle rotation
- **Concurrent execution**: If agent runs multiple tasks, log may have multiple "complete" messages
- **Timezone consistency**: Timestamps should use consistent timezone (UTC recommended)
- **File permissions**: Script needs read access to agent log files

**Edge Cases:**
- Empty log files (no entries yet)
- Log files with only whitespace
- Very recent completion (race condition between script finish and status check)
- Network filesystem delays (log not yet flushed)

**Future Enhancements:**
- Add `--watch` mode to monitor status changes in real-time
- Add agent health metrics (uptime, task count, avg duration)
- Integrate with `scripts/attach.sh` to show status in selection menu
- Add metric output: `--format=prometheus` for monitoring systems
